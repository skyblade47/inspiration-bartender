## 原则零：功能对齐检查 — 所有改动必须符合产品定位

### 0.1 核心理念

> **任何代码修改、功能设计、架构调整，必须首先通过产品定位一致性检查。发现偏离时必须取得用户确认才能继续推进。**

### 0.2 产品核心定位（不可偏离）

| 维度 | 定义 | 来源 |
|------|------|------|
| **产品名称** | 灵感调酒师（Inspiration Bartender） | `README.md` |
| **产品定位** | 以"调酒"为隐喻的灵感收集与完善工具 | `README.md` |
| **核心哲学** | 用户输入灵感 → LLM 辅助完善 → 多灵感碰撞产生新想法 | `README.md` |
| **核心隐喻** | 灵感 = 液体 / 杯子 = 容器 / 碰撞 = 摇晃混合 | `specs/glass-component.md` |
| **技术栈** | React Native (Expo) + TypeScript + Zustand + Expo SQLite | `README.md` |
| **数据流向** | 桌面宠物 → 灵感调酒师 → AI 写作教练（单向同步） | `specs/sync-system.md` |

### 0.3 明确禁止的方向（除非用户明确批准）

| 禁止方向 | 冲突原则 | 如需推进 |
|---------|---------|---------|
| 脱离"调酒"隐喻的功能改造 | 违背核心产品视觉语言 | 用户书面确认 |
| 将灵感调酒师改造成纯文本编辑器 | 偏离"调酒"交互定位 | 用户书面确认 |
| 引入与桌面宠物/写作教练无关的独立功能 | 违背单向同步数据流 | 用户书面确认 |
| 重大技术栈迁移（如 RN → 纯原生） | 违背现有技术选择 | 用户书面确认 |
| 未经设计直接修改核心模块代码 | 违背"设计先行"原则 | 用户书面确认 |

### 0.4 功能对齐检查清单（每次开发任务前必须完成）

```
□ 这项改动是否符合"调酒隐喻"的核心定位？
  → 是否保持杯子 / 液体 / 碰撞的视觉一致性？
  → 用户交互是否仍以"调酒"为核心隐喻？

□ 这项改动是否符合单向同步数据流？
  → 数据从桌面宠物流入灵感调酒师，再流向写作教练
  → 是否避免反向依赖或双向同步复杂化？

□ 这项改动是否在灵感调酒师的职责边界内？
  → 灵感收集 / 完善 / 碰撞 → 是
  → 写作辅助 / 记忆管理 → 交给写作教练
  → 系统托盘 / 提醒 → 交给桌面宠物

□ 技术选型是否与现有栈一致？
  → React Native Expo（不引入原生模块）
  → Zustand（不引入 Redux/MobX）
  → Expo SQLite（不引入其他数据库）

□ 如以上任何一项回答为"否"或"不确定"：
  → ⚠️ 立即暂停，向用户说明偏离点
  → 等待用户明确确认后方可继续
```

### 0.5 偏离确认流程

```
检测到可能偏离 → 整理偏离分析 → 提交用户确认 → 根据用户决策执行
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
       批准推进    修改方案     否决放弃
          │           │           │
     记录决策    重新对齐    记录决策
     继续工作    再次检查    关闭任务
```

**偏离分析报告格式**：

```markdown
## 功能对齐偏离提醒

**任务**: [任务名称]
**偏离点**: 具体哪些设计不符合产品定位
**风险等级**: 高/中/低
**影响范围**: 可能影响哪些现有功能或原则
**替代方案**: 如果有更符合定位的实现方式
**建议**: 推进 / 修改方案 / 放弃

请用户确认决策。
```

## 原则一：设计方案 + Ponytail 指导代码设计

### 1.1 核心理念

> **设计先行，代码后行。用 Ponytail 的"最懒资深工程师"思维写代码——最好的代码是你没写过的代码。**

### 1.2 Ponytail 在灵感调酒师中的应用

我们将 Ponytail 的精简哲学作为**代码设计第一准则**：

```
需求分析 → 设计方案 → Ponytail精简 → 设计评审 → 编码实现 → 验证收尾
   │           │            │              │           │           │
   └─ 要做什么  └─ 怎么做    └─ 能更简单吗  └─ 这样对吗  └─ 最少代码 └─ 验证通过
```

**Ponytail 精简检查清单**（设计评审阶段使用）：

```
□ 这个功能是否真的需要？能否用现有功能组合实现？
□ 能否用更少的代码实现同样效果？
□ 是否引入了不必要的抽象层？（为未来写代码是大忌）
□ 能否复用现有组件/服务而非新建？
□ 新增的依赖是否真的必要？（npm 包引入需谨慎）
□ 能否用一个函数替代一个类/模块？
□ 能否用配置驱动替代硬编码？
□ 是否符合 "Expo 原生" 原则，避免原生模块？
```

### 1.3 五阶段详解

| 阶段 | 产出物 | 检查点 |
|------|--------|--------|
| **1. 需求分析** | 需求文档（简洁要点） | 目标清晰、范围明确 |
| **2. 设计方案** | 技术方案 + 架构图 + 组件树 | 接口定义、数据流、组件树 |
| **3. Ponytail 精简** | 精简后的方案 | **最少代码、最少文件、最少依赖** |
| **4. 设计评审** | 评审记录 + 修改意见 | 可行性、一致性、影响范围 |
| **5. 编码实现** | 代码 + 类型检查 | 遵循精简方案 |
| **6. 验证收尾** | 检查报告 | 类型检查通过 / 功能正常 |

### 1.4 设计文档模板（灵感调酒师专用）

**设计方案必须包含**：

```markdown
# 设计文档 - [模块/功能名称]

## 1. 功能概述
一句话说清楚做什么，以及为什么需要它

## 2. 涉及文件清单
| 文件 | 操作 | 说明 |
|------|------|------|
| `src/path/to/file.ts` | 新增/修改 | 简要说明 |

## 3. 类型/数据模型
[TypeScript 接口/类型定义]

## 4. 组件树 / 数据流图
[简要的组件层级或数据流向说明]

## 5. 核心逻辑说明
[关键实现思路，不超过 3 段]

## 6. 边界情况处理
□ 空状态 / □ 错误状态 / □ 加载状态 / □ 并发冲突

## 7. 验收标准
- [ ] TypeScript 类型检查通过（0 错误）
- [ ] 核心功能正常
- [ ] 符合 CODE_STANDARDS.md
- [ ] 文档已更新（如有需要）

## 8. 风险评估
风险等级：低 / 中 / 高
说明：...
```

### 1.5 设计阶段禁止事项

- ❌ 跳过设计方案直接写代码
- ❌ 设计方案中没有组件树/数据流就开始编码
- ❌ 不经过用户确认就进入编码阶段
- ❌ 编码时随意偏离设计方案（如需偏离，先更新方案并重新确认）

## 原则二：Subagent 编写 + Subagent 检查（写→审→改→查闭环）

### 2.1 核心理念

> **同一个 AI 不能既当运动员又当裁判。编写和检查必须由不同的 subagent 完成。审查者只给意见不直接修改，形成写→审→改→查的完整闭环。**

### 2.2 四轮闭环审查流程

```
                        ┌──────────────────────────────────────┐
                        │          第一轮：编写 (Write)          │
                        │                                      │
                        │  ┌─────────────────────────────────┐ │
                        │  │  Coder Subagent (编写者)         │ │
                        │  │  - 实现功能                     │ │
                        │  │  - 遵循设计方案                 │ │
                        │  │  - 添加注释                     │ │
                        │  └──────────────┬──────────────────┘ │
                        │                 │ 产出：代码         │
                        └─────────────────┼────────────────────┘
                                          ▼
                        ┌──────────────────────────────────────┐
                        │         第二轮：审查 (Review)         │
                        │                                      │
                        │  ┌─────────────────────────────────┐ │
                        │  │  Reviewer Subagent (审查者)      │ │
                        │  │  - 逐条检查代码                 │ │
                        │  │  - ⚠️ 只给修改意见，不动代码    │ │
                        │  │  - 输出审查报告                 │ │
                        │  └──────────────┬──────────────────┘ │
                        │                 │ 产出：审查意见     │
                        └─────────────────┼────────────────────┘
                                          ▼
                              ┌───────────────────┐
                              │  审查是否通过？    │
                              └───────┬───────┬───┘
                                 通过 │       │ 不通过
                                      │       ▼
                                      │ ┌──────────────────────────────────────┐
                                      │ │         第三轮：修复 (Fix)           │
                                      │ │                                      │
                                      │ │  ┌─────────────────────────────────┐ │
                                      │ │  │  Coder Subagent (编写者)         │ │
                                      │ │  │  - 根据审查意见逐条修复         │ │
                                      │ │  │  - 标注修复内容                 │ │
                                      │ │  └──────────────┬──────────────────┘ │
                                      │ │                 │ 产出：修复代码     │
                                      │ └─────────────────┼────────────────────┘
                                      │                   │
                                      │                   ▼
                                      │ ┌──────────────────────────────────────┐
                                      │ │         第四轮：复查 (Verify)        │
                                      │ │                                      │
                                      │ │  ┌─────────────────────────────────┐ │
                                      │ │  │  Reviewer Subagent (审查者)      │ │
                                      │ │  │  - 逐条验证修复结果             │ │
                                      │ │  │  - ⚠️ 仍只给意见，不动代码      │ │
                                      │ │  │  - 输出复查报告                 │ │
                                      │ │  └──────────────┬──────────────────┘ │
                                      │ │                 │                    │
                                      │ └─────────────────┼────────────────────┘
                                      │                   │
                                      │      ┌────────────┴────────────┐
                                      │      ▼                         ▼
                                      │   修复通过                  修复不通过
                                      │      │                         │
                                      └──────┘              (返回第三轮，最多3次)
                                                                     │
                                                                     ▼
                                                                超过3次上报
                                                                Main Agent 介入
                                      │
                                      ▼
                        ┌──────────────────────────────────────┐
                        │         Main Agent 最终确认           │
                        │  - 合并结果                          │
                        │  - 记录经验                          │
                        │  - 更新错误库                        │
                        └──────────────────────────────────────┘
```

### 2.3 核心铁律：审查者不直接修改代码

> ⚠️ **审查 subagent 永远只输出审查意见，绝不直接修改代码。**

| 审查者可以 | 审查者不可以 |
|-----------|-------------|
| ✅ 指出具体问题位置（文件+行号） | ❌ 直接修改任何代码文件 |
| ✅ 说明问题原因和影响 | ❌ 运行写文件/写代码操作 |
| ✅ 给出修复建议（可附示例代码片段） | ❌ 以"帮你改好了"的方式操作 |
| ✅ 标注严重程度（严重/中等/轻微） | ❌ 在意见中使用模糊表述（"可能有问题"） |
| ✅ 在复查时逐条对照确认修复 | ❌ 复查时发现新问题直接修改 |

### 2.4 Subagent 分工规范

#### Coder Subagent（编写者）职责

```
Coder 必须做到：
□ 严格遵循设计方案
□ 遵循 docs/CODE_STANDARDS.md 所有规范
□ 添加中文注释（复杂逻辑处）
□ 处理所有边界情况（空值、错误、加载中）
□ 保持变更范围最小化（不修改与任务无关的文件）
□ 修复时逐条标注对应哪个审查意见
□ 完成后运行 `npx tsc --noEmit` 确保类型检查通过
```

#### Reviewer Subagent（审查者）职责

```
审查检查清单：
□ TypeScript 类型定义是否完整（无隐式 any、无 `// @ts-ignore`）
□ React 组件 Props 接口是否完整
□ useEffect 依赖数组是否正确（无 ESLint 警告）
□ 异步操作 / 定时器是否正确清理（资源泄漏检查）
□ 错误处理是否覆盖所有分支
□ 是否引入不必要的依赖 / npm 包
□ 是否修改了与任务无关的文件
□ 代码风格是否符合项目规范（CODE_STANDARDS.md）
□ 组件命名是否符合现有约定
```

**审查报告格式**：

```markdown
# 审查报告 - [任务名称]

## 总体评价
[通过 / 需要修复]

## 问题列表

### [严重] 问题1：xxx
- **位置**: `path/file.ts:行号`
- **问题**: 具体描述
- **影响**: 可能导致的后果
- **建议**: 修复方案（可附示例代码片段）

### [中等] 问题2：xxx
- ...

### [轻微] 问题3：xxx
- ...
```

### 2.5 循环上限与上报机制

- **修复-复查循环上限**：最多 3 轮
- **超过 3 轮**：上报 Main Agent，由主 Agent 决策（可能重新设计或放弃当前方案）
- **审查者责任**：永远客观，不"放水"，也不"过度审查"

## 原则三：错误库建立

### 3.1 核心理念

> **同样的错误不犯第二次。每个错误都是团队的资产。**

### 3.2 错误库结构

```
docs/errors/
├── index.md                      # 错误索引（总览 + 分类统计）
├── template.md                   # 新错误记录模板
├── ERR-001-类型错误.md           # 历史错误
├── ERR-002-Git推送失败.md
└── ...
```

### 3.3 错误记录模板

每条错误记录必须包含：

```markdown
# [ERR-XXX] 错误标题

| 字段 | 内容 |
|------|------|
| **错误编号** | ERR-XXX |
| **发现日期** | YYYY-MM-DD |
| **严重程度** | 🔴 致命 / 🟠 高 / 🟡 中 / 🟢 低 |
| **涉及模块** | 玻璃杯 / 同步系统 / LLM 集成 / 评分 / 碰撞 |
| **涉及文件** | `path/to/file.ts` |
| **发现阶段** | 设计 / 编码 / 审查 / 验收 |
| **状态** | ⏳ 待处理 / 🔧 处理中 / ✅ 已解决 / 🚫 已关闭 |

## 错误描述
一句话描述错误现象 + 复现步骤

## 根本原因
根因分析（为什么会发生）

## 解决方案
具体修复步骤 + 代码变更摘要

## 预防措施
如何防止再次发生（检查清单、自动化检查、文档提醒）

## 关联文档
- [设计文档](specs/xxx.md) - 相关设计
- [原则对照](WORK_PRINCIPLES.md#原则x) - 违反了哪条原则
```

### 3.4 错误库使用流程

```
发现错误 → 记录到错误库 → 分析根因 → 修复代码 → 更新预防措施
                                              ↓
                                        审查 Checklist 更新
                                              ↓
                                    加入 docs/errors/index.md
```

### 3.5 常见错误分类（灵感调酒师特定）

| 分类 | 典型错误 | 预防措施 |
|------|---------|---------|
| **资源泄漏** | useEffect 未清理、定时器未销毁 | Reviewer checklist 专项检查 |
| **类型缺失** | 隐式 any、Props 不完整 | `npx tsc --noEmit` 阻断提交 |
| **状态管理** | Zustand store 设计不合理 | 参考 specs/*.md 中状态设计 |
| **同步协议** | 与桌面宠物/写作教练的同步协议不一致 | 对照 specs/sync-system.md |
| **边界遗漏** | 空状态、错误状态、加载状态未处理 | 设计阶段强制列出边界 |
| **依赖膨胀** | 引入不必要的 npm 包 | Reviewer 专项检查 |
| **原生模块** | 引入非 Expo 原生依赖 | 原则零技术选型检查 |

### 3.6 错误库 → 自动化检查

目标：将错误库中的常见错误转化为自动化检查

```
错误库记录 → 分析模式 → 添加 ESLint 规则 / TypeScript 约束 → CI 检查
```

灵感调酒师当前 CI 检查：
- `npx tsc --noEmit` —— 类型检查（必须 0 错误）
- 后续可扩展：ESLint、Prettier、自定义规则脚本

## 原则四：可复用代码库 + 代码参考库

### 4.1 核心理念

> **不要重复造轮子。写好一次，到处使用。参考优质代码，站在巨人肩膀上。**

### 4.2 代码资产双层架构

```
┌─────────────────────────────────────────────────────────────┐
│                      代码资产体系                             │
│                                                             │
│  ┌─────────────────────┐    ┌─────────────────────────────┐ │
│  │   可复用代码库       │    │     代码参考库              │ │
│  │ (Reusable Code)    │    │  (Reference Code)            │ │
│  │                     │    │                             │ │
│  │  直接复用的代码     │    │  参考学习的代码             │ │
│  │  - 工具函数         │    │  - 开源项目优秀模式         │ │
```

## 复检模式：换设备/换场景继续开发

### 核心理念

> **所有工作状态、模式、规范全部文档化。换一台设备，读一遍文档，就能无缝继续。**

### 新场景启动流程（5 分钟上手）

```
新设备拉取代码
    │
    ▼
第一步：读核心文档（按优先级）
  ┌─────────────────────────────────────────────┐
  │ 1. docs/WORK_PRINCIPLES.md    ← 先看原则    │
  │ 2. docs/index.md              ← 文档索引    │
  │ 3. docs/WORKFLOW.md           ← 执行流程    │
  │ 4. docs/errors/index.md       ← 不踩坑      │
  │ 5. docs/CODE_STANDARDS.md     ← 编码规范    │
  │ 6. docs/specs/*.md            ← 模块规范    │
  └─────────────────────────────────────────────┘
    │
    ▼
第二步：环境准备
  npm install
  npx tsc --noEmit              # 确认类型检查通过
    │
    ▼
第三步：确认当前进度
  打开 docs/index.md → 查看当前项目状态 → 确定下一个待办任务
    │
    ▼
第四步：开始工作
  遵循五大原则执行任务
```

### 任务完成后必须更新

每次完成一个任务模块后，**立即**更新以下文档：

| 必须更新 | 更新内容 | 负责者 |
|---------|---------|-------|
| **docs/index.md** | 标记模块状态、更新进度 | Main Agent |
| **docs/specs/xxx.md** | 如有设计调整，同步更新设计文档 | Main Agent |
| **docs/errors/index.md** | 记录新发现的错误及预防措施 | Main Agent |
| **WORK_PRINCIPLES.md** | 如有流程优化则更新版本 | Main Agent |

### Git 提交规范

完成任务并更新文档后，提交信息格式：

```
feat/fix/docs(scope): 简短描述

- 具体变更 1
- 具体变更 2

关联: docs/specs/xxx.md / docs/errors/ERR-xxx.md
```

scope 可选值：`glass` / `sync` / `llm` / `scoring` / `collision` / `docs`

## 快速检查清单

### 每次开发任务开始前

```
□ 是否通过了功能对齐检查？（原则零）
□ 若有偏离产品定位，是否已取得用户确认？（原则零）
□ 是否有设计方案（docs/specs/*.md）？（原则一）
□ 设计方案是否经过用户确认？（原则一）
□ 是否计划使用 Coder + Reviewer 双 subagent？（原则二）
□ 是否查阅了错误库中的相关记录？（原则三）
□ 是否查阅了现有可复用代码？（原则四）
```

### 每次开发任务完成后

```
□ npx tsc --noEmit 通过（0 错误）
□ 是否更新了相关设计文档？（原则一）
□ 是否更新了错误库（如有新错误）？（原则三）
□ 是否有新的可复用代码需要沉淀？（原则四）
□ Reviewer 的检查清单是否需要更新？（原则二+三）
□ Git 提交并推送
```
