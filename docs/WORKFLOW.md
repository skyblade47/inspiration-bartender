## 核心流程概述

所有开发和修改任务都必须遵循以下流程：

```
设计阶段 → 修改阶段 → 验收阶段
   ↓          ↓          ↓
 用户确认   用户确认   用户确认
```

**重要**：每个阶段完成后都必须等待用户确认，才可以进入下一阶段。

> ⚠️ **所有阶段都必须遵循 [WORK_PRINCIPLES.md](WORK_PRINCIPLES.md) 五大原则。**
> - 原则零：开始前确认功能对齐
> - 原则一：设计方案必须包含类型/边界/Ponytail 精简
> - 原则二：写→审→改→查 闭环（Coder + Reviewer 双 Agent）
> - 原则三：发现错误立即记录到 `docs/errors/ERR-XXX.md`
> - 原则四：复用现有代码，避免重复造轮子

## 🔷 第一阶段：设计阶段

### 目标
- 分析需求并理解问题
- **通过功能对齐检查**（原则零）—— 确认改动符合产品定位
- 确定技术方案，遵循 **Ponytail 精简原则**（原则一）
- 创建详细的设计文档
- 确保方案可行且符合项目标准

### 执行者
- **设计 Agent**（由主 Agent 委派）

### ⚠️ 原则零：功能对齐检查（最先执行）

**在开始任何设计工作之前，必须先完成此检查清单**：

```
□ 这项改动是否符合"调酒隐喻"的核心定位？
  → 是否保持杯子 / 液体 / 碰撞的视觉一致性？
  → 用户交互是否仍以"调酒"为核心隐喻？

□ 这项改动是否符合单向同步数据流？
  → 数据从桌面宠物流入灵感调酒师，再流向写作教练
  → 是否避免反向依赖或双向同步复杂化？

□ 这项改动是否在灵感调酒师的职责边界内？
  → 灵感收集 / 完善 / 碰撞 → 是
  → 写作辅助 / 记忆管理 → 交给写作教练

□ 技术选型是否与现有栈一致？
  → React Native Expo / Zustand / Expo SQLite

□ 如以上任何一项回答为"否"或"不确定"：
  → ⚠️ 立即暂停，向用户说明偏离点
  → 等待用户明确确认后方可继续
```

### 工作步骤

#### 1. 需求分析（预计 15-30 分钟）
- [ ] 阅读并理解用户需求
- [ ] 检查现有代码和文档
- [ ] 确定需要修改的模块
- [ ] 识别潜在的依赖关系和风险
- [ ] 记录需要解决的关键问题
- [ ] **查阅 `docs/errors/index.md` 确认是否有相关历史错误**

#### 2. 方案设计（预计 30-60 分钟）
- [ ] 设计数据结构和数据模型（TypeScript interface）
- [ ] 设计组件或模块结构
- [ ] 设计 API 接口（如有）
- [ ] 设计状态管理方案（如有）
- [ ] 设计用户交互流程（如有）
- [ ] **Ponytail 精简检查**：
  ```
  □ 这个功能是否真的需要？能否用现有功能组合实现？
  □ 能否用更少的代码实现同样效果？
  □ 是否引入了不必要的抽象层？
  □ 能否复用现有组件/服务而非新建？
  □ 新增的依赖是否真的必要？
  □ 能否用配置驱动替代硬编码？
  ```

#### 3. 风险评估
- [ ] 识别技术风险
- [ ] 评估实现难度
- [ ] 确定可能的问题点
- [ ] 提出预防措施
- [ ] **列出所有边界情况**（空状态 / 错误状态 / 加载状态 / 组件卸载清理）

#### 4. 创建设计文档
- [ ] 编写设计规范（位置：`docs/specs/[module-name].md`）
- [ ] 包含 TypeScript 数据模型定义
- [ ] 包含组件树 / 数据流图
- [ ] 包含验收标准
- [ ] 包含测试用例（可选）

### 设计文档模板（使用 WORK_PRINCIPLES.md 规定模板）

```markdown
# 设计文档 - [模块/功能名称]

## 1. 功能概述
一句话说清楚做什么，以及为什么需要它

## 2. 涉及文件清单
| 文件 | 操作 | 说明 |
|------|------|------|
| `src/path/to/file.ts` | 新增/修改 | 简要说明 |

## 3. 类型/数据模型
[TypeScript 接口/类型定义]

## 4. 组件树 / 数据流图
[简要的组件层级或数据流向说明]

## 5. 核心逻辑说明
[关键实现思路，不超过 3 段]

## 6. 边界情况处理
□ 空状态 / □ 错误状态 / □ 加载状态 / □ 并发冲突

## 7. 验收标准
- [ ] npx tsc --noEmit 通过（0 错误）
- [ ] 核心功能正常
- [ ] 符合 CODE_STANDARDS.md
- [ ] 文档已更新（如有需要）

## 8. 风险评估
风险等级：低 / 中 / 高
```

### 输出文档
- **模块设计规范**：`docs/specs/[module-name].md`
- **更新项目索引**：更新 `docs/index.md`

### 设计完成后需要提供
1. 清晰的设计方案描述
2. 预期的文件变更列表
3. 验收标准（测试用例）
4. 潜在的风险和问题
5. **原则零检查结果**：是否涉及功能偏离？

### 用户确认点
在进入修改阶段之前，必须确认：
- [ ] **原则零功能对齐检查通过**（无偏离或有用户书面确认）
- [ ] 用户理解设计方案
- [ ] 用户同意技术选型
- [ ] 用户确认验收标准
- [ ] 用户没有其他重大修改要求

## 🔷 第二阶段：修改阶段

### 目标
- 根据设计文档实现代码
- 遵循 **Coder + Reviewer 双 Agent 写→审→改→查 闭环**（原则二）
- 确保代码符合项目标准
- 确保功能按预期工作
- 保持代码质量和可维护性

### 执行者
- **修改 Agent (Coder)**：负责编写代码
- **验收 Agent (Reviewer)**：负责审查代码，**只给意见不直接修改**

### ⚠️ 原则二：写→审→改→查 闭环（核心铁律）

> ⚠️ **审查者永远只输出审查意见，绝不直接修改代码。**

#### Coder Subagent 必须做到
```
□ 严格遵循设计方案
□ 遵循 docs/CODE_STANDARDS.md 所有规范
□ 添加中文注释（复杂逻辑处）
□ 处理所有边界情况（空值、错误、加载中）
□ 保持变更范围最小化（不修改与任务无关的文件）
□ 修复时逐条标注对应哪个审查意见
□ 完成后运行 npx tsc --noEmit 确保类型检查通过
```

#### Reviewer Subagent 必须做到
```
审查检查清单：
□ TypeScript 类型定义是否完整（无隐式 any、无 // @ts-ignore）
□ React 组件 Props 接口是否完整
□ useEffect 依赖数组是否正确（无 ESLint 警告）
□ 异步操作 / 定时器是否正确清理（资源泄漏检查）
□ 错误处理是否覆盖所有分支
□ 是否引入不必要的依赖 / npm 包
□ 是否修改了与任务无关的文件
□ 代码风格是否符合项目规范（CODE_STANDARDS.md）
□ 组件命名是否符合现有约定
□ 边界情况是否在代码中得到处理（空状态/错误状态/卸载清理）
□ 是否查阅了 docs/errors/index.md 中的相关历史错误
□ 是否遵循了 WORK_PRINCIPLES.md 规定的命名和注释规范
```

### 工作步骤

#### 1. 代码实现（Coder 执行）
- [ ] 严格遵循 `docs/specs/[module-name].md` 设计文档
- [ ] 创建或修改 TypeScript 文件
- [ ] 确保 TypeScript 类型定义正确（无隐式 any）
- [ ] 实现核心逻辑
- [ ] 实现用户界面（如有）
- [ ] **处理所有边界情况**（参考设计文档第 6 节）
- [ ] 添加中文注释（复杂逻辑处）
- [ ] **遵循 CODE_STANDARDS.md 命名规范**
- [ ] 集成到现有系统

#### 2. 代码自检（Coder 执行）
- [ ] 运行 `npx tsc --noEmit` —— 必须 0 错误
- [ ] 检查是否有类型警告
- [ ] 确保代码符合 `CODE_STANDARDS.md`
- [ ] 确认未修改与任务无关的文件

#### 3. 代码审查（Reviewer 执行）
- [ ] 逐条执行上述 Reviewer checklist
- [ ] **不修改任何代码，只输出审查意见**
- [ ] 标注每个问题的严重程度（严重 / 中等 / 轻微）
- [ ] 给出修复建议（可附示例代码片段）

#### 4. 审查结果处理
- [ ] **审查通过** → 进入步骤 6（Git 提交）
- [ ] **审查不通过** → Coder 根据审查意见逐条修复 → 返回步骤 3 复查
- [ ] **循环上限**：最多 3 轮修复-复查，超过则上报 Main Agent

#### 5. 发现错误时（原则三）
- [ ] 立即创建 `docs/errors/ERR-XXX.md`（参考 errors/index.md 模板）
- [ ] 分析根因并写入错误记录
- [ ] 修复代码后更新错误记录状态为"已解决"
- [ ] 将预防措施加入 Reviewer checklist

#### 6. Git 提交
- [ ] 使用有意义的提交信息（遵循 commit message 规范）
- [ ] 遵循 scope：`glass` / `sync` / `llm` / `scoring` / `collision` / `docs`
- [ ] **提交前务必 `npx tsc --noEmit` 确认通过**

### 输出
- 代码变更（在相应的源文件中）
- 审查报告（如审查发现问题）
- 如有新错误：错误记录（`docs/errors/ERR-XXX.md`）

### 修改完成后需要提供
1. 修改的文件列表
2. `npx tsc --noEmit` 结果（截图或文本）
3. 基本功能测试结果
4. 遇到的问题和解决方案（如有）
5. 审查报告（如有）

### 用户确认点
在进入验收阶段之前，必须确认：
- [ ] `npx tsc --noEmit` 通过（0 错误）
- [ ] Coder + Reviewer 双 Agent 闭环已完成
- [ ] 用户理解已实现的功能
- [ ] 用户确认代码已成功提交
- [ ] 用户没有其他修改要求
- [ ] 用户确认可以开始验收

## 🔷 第三阶段：验收阶段

### 目标
- 验证功能是否符合设计文档
- 确保没有引入新问题
- 确认代码质量（TypeScript + 风格）
- 记录成功或失败
- **维护错误库**（原则三）

### 执行者
- **验收 Agent (Reviewer)**（由主 Agent 委派）

### 工作步骤

#### 1. 功能验证（预计 15-30 分钟）
- [ ] 验证核心功能是否正常工作
- [ ] 验证边缘情况和边界条件
- [ ] 验证错误处理是否正常
- [ ] 验证集成是否正常（如适用）

#### 2. 代码审查（最终复查）
- [ ] 再次检查代码是否符合 `CODE_STANDARDS.md`
- [ ] 检查类型定义是否完整
- [ ] 检查注释和文档是否充分
- [ ] 检查代码组织和结构是否合理

#### 3. 验证测试
- [ ] 运行 `npx tsc --noEmit` —— **必须 0 错误**
- [ ] 手动验证关键路径
- [ ] 验证无回归（其他模块正常）

#### 4. 结果记录（原则三 + 原则四）
- [ ] 记录验收结果
- [ ] 记录发现的问题（如有）
- [ ] 记录解决的问题（如有）
- [ ] **如有新错误：立即创建 `docs/errors/ERR-XXX.md`**
- [ ] **如有可复用代码：考虑沉淀到 `src/services/` 或 `src/shared/`**

### 验收标准

**必须全部满足：**
- [ ] `npx tsc --noEmit` 类型检查通过（0 错误）
- [ ] 核心功能正常工作
- [ ] 代码符合 `CODE_STANDARDS.md`
- [ ] 文档已更新

**建议满足：**
- [ ] 有中文注释解释复杂逻辑
- [ ] 边界情况处理正常
- [ ] 错误处理优雅

### 可能的结果
1. **✅ 通过** - 所有标准都满足，任务完成
2. **⚠️ 有条件通过** - 需要小幅修改，将要求返回修改阶段
3. **❌ 未通过** - 需要重大修改，将要求返回设计阶段或重做

### 验收完成后
如果验收通过：
- [ ] 更新 `docs/index.md`（如有里程碑更新）
- [ ] 记录成功
- [ ] 准备进入下一个任务或等待用户新需求

如果验收未通过：
- [ ] 记录失败原因（在 `docs/errors/`）
- [ ] 提供修复建议
- [ ] 返回适当的阶段重新执行

### 用户确认点
在任务最终完成之前，必须确认：
- [ ] 用户确认功能正常工作
- [ ] 用户确认代码质量
- [ ] 用户没有其他修改要求
- [ ] 用户确认任务完成

## 🔶 错误处理和历史记录

### 错误分类
| 严重程度 | 说明 | 示例 |
|----------|------|------|
| 🔴 致命 | 阻止功能正常工作 | 应用崩溃，功能完全不可用 |
| 🟠 高 | 主要功能受损 | 功能不完整，重要流程失败 |
| 🟡 中 | 功能部分可用但有问题 | 边缘情况失败，性能问题 |
| 🟢 低 | 小问题，不影响核心功能 | 样式问题，小错误，文档遗漏 |

### 错误处理流程
1. **记录错误** - 创建 `docs/errors/ERR-XXX.md`
2. **分析原因** - 确定根本原因
3. **提出解决方案** - 说明如何修复
4. **执行修复** - 如果问题在当前范围内，尝试修复
5. **验证修复** - 确保问题已解决
6. **更新文档** - 记录最终解决方案

### 错误记录格式
```markdown
# ERR-XXX: 错误标题

## 基本信息
- 发现日期：YYYY-MM-DD
- 严重程度：🔴致命/🟠高/🟡中/🟢低
- 状态：待处理/处理中/已解决

## 问题描述
详细描述问题现象

## 复现步骤
1. ...
2. ...

## 根本原因
为什么会出现这个问题

## 解决方案
如何修复这个问题

## 预防措施
如何防止再次发生
```

## 🔶 文档管理指南

### 文档命名规范
- **设计文档**：`docs/specs/[module-name].md`
- **阶段计划**：`docs/plans/Phase[X]-[description].md`
- **错误记录**：`docs/errors/ERR-[XXX]-[description].md`
- **参考文档**：`docs/references/[topic].md`

### 文档创建规则
1. 每个新模块或重大修改都应该有一个设计文档
2. 设计文档应该在设计阶段完成并由用户确认
3. 所有文档都应该包含版本号和更新日期
4. 文档应该清晰、结构化、易于理解

### 文档维护
- **设计文档**：应该反映当前实现状态
- **错误历史**：应该持续更新，记录解决的每个问题
- **参考文档**：应该定期更新以反映API变更
- **索引文档**：`docs/index.md` 应该保持最新

## 🔶 版本控制和 Git 规范

### Git Commit Message 格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### 类型（type）
- `feat` - 新功能
- `fix` - 修复bug
- `docs` - 文档修改
- `style` - 样式调整（不影响代码逻辑）
- `refactor` - 重构（不改变功能）
- `test` - 添加或修改测试
- `chore` - 构建过程或辅助工具

### 范围（scope）
- `glass` - 玻璃杯组件
- `sync` - 同步系统
- `llm` - LLM集成
- `scoring` - 评分系统
- `collision` - 碰撞系统
- `docs` - 文档

### 示例
```
feat(glass): 添加8种玻璃杯子类型和动画效果

- 实现白兰地杯、香槟杯、红酒杯等8种杯子样式
- 添加液体填充动画
- 完善响应式布局

Fixes: #123
```

## 🔶 子 Agent 执行指南

### 子 Agent 的职责
1. **专注于单一任务** - 不要尝试做超出范围的事情
2. **保持清晰的沟通** - 每完成一步都要清晰地告知用户
3. **遵循文档规范** - 所有工作都要记录在适当的文档中
4. **尊重用户确认** - 每个阶段完成后都要等待用户确认
5. **遇到问题要记录** - 将问题记录在错误历史中，无论是否能解决（原则三）

### 子 Agent 的工作流程
1. 接受任务（由主 Agent 委派）
2. 阅读相关文档（WORK_PRINCIPLES.md、设计文档、代码标准）
3. **先执行原则零检查** - 确认功能对齐
4. **执行设计/修改/验收流程**
5. 输出结果
6. 请求用户确认
7. 等待用户响应
8. 根据用户响应继续或停止

### 子 Agent 之间的协作
- **设计 Agent** 完成后将设计文档交给**修改 Agent (Coder)**
- **Coder** 完成后交给**验收 Agent (Reviewer)** 进行审查
- **Reviewer** 输出审查报告（只给意见，不修改），Coder 修复后复查
- **验收 Agent** 完成后报告给 Main Agent 和用户
- **Main Agent** 维护错误库和代码库（原则三 + 原则四）

```
 需求确认    创建设计    确认设计    实现代码    确认变更   测试验收
```

### Bug修复流程
```
问题报告 → 错误记录 → 设计（解决方案）→ [用户确认] → 修改（修复）→ [用户确认] → 验收 → [用户确认] → 完成
```

### 文档更新流程
```
文档需求 → 设计（结构）→ [用户确认] → 修改（编写）→ [用户确认] → 完成
```

---

## 🔶 常见问题（FAQ）

### Q: 如果用户在确认后又要求回到前面的阶段怎么办？
A: 这是允许的。记录用户的新要求，更新相应文档，然后重新执行适当的阶段。重要的是确保用户需求得到满足。

### Q: 如果遇到无法解决的问题怎么办？
A: 详细记录问题在错误历史中（`docs/errors/ERR-XXX.md`），诚实地告知用户，提出替代方案，等待用户决定下一步。

### Q: 如果设计文档和实际实现不一致怎么办？
A: 更新设计文档以反映实际实现，设计文档应该是当前系统状态的准确反映，在修改阶段结束时更新。

### Q: 如何处理小的修改（例如单个小函数修改）？
A: 仍然遵循设计-修改-验收流程，但可以简化。设计阶段可以是简短的要点列表，而不是完整的设计文档。核心原则是每个阶段都要有用户确认。
