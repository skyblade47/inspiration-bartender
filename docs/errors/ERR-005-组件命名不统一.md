# [ERR-005] 组件命名风格不统一 / 路径混乱

| 字段 | 内容 |
|------|------|
| **错误编号** | ERR-005 |

| **严重程度** | 🟢 低 |
| **涉及模块** | 规范 / 代码组织 |
| **涉及文件** | `src/components/*/`、`src/screens/*/`、`src/services/*/` |
| **发现阶段** | 审查阶段 / Code Review |
| **状态** | ✅ 已解决 |

---

## 错误现象

代码库中存在命名和路径风格不一致的问题：

- 组件文件名有的用 PascalCase（`GlassComponent.tsx`）、有的用 kebab-case（`glass-component.tsx`）
- 目录命名有的全小写（`components/glass/`）、有的大小写混合（`Components/Glass/`）
- 服务文件有的用 index 导出（`llm/index.ts`）、有的直接导出（`llm.ts`）
- 导入时大小写不一致导致 Linux/macOS 构建失败（Windows 不敏感）

**预期行为**：代码风格统一，路径遵循约定俗成，可预测。  
**实际行为**：协作者需要猜测文件位置，新成员上手成本高，跨平台构建可能失败。

## 复现步骤

1. 在 `src/components/` 目录下列出所有文件，观察命名模式
2. 搜索 import 语句，看是否有不一致的路径写法
3. 在 Linux/macOS 上尝试编译（如果开发在 Windows 上），观察到路径错误

---

## 根本原因分析

| 常见子类 | 说明 |
|---------|------|
| **无 CODE_STANDARDS** | 项目初期未建立明确的命名规范文档 |
| **跨平台意识缺失** | Windows 文件系统大小写不敏感，但最终打包在 Linux 上运行 |
| **不同 subagent 产出** | 多个 subagent 独立编写时未遵循同一套命名约定 |

深层原因：原则一（设计方案）阶段缺少"命名规范"检查；原则四（可复用代码库）缺少统一的目录结构约定。

---

## 解决方案

### 统一命名规范（已写入 `docs/CODE_STANDARDS.md`）

| 对象 | 规范 | 示例 |
|------|------|------|
| **React 组件文件** | PascalCase + `.tsx` | `GlassComponent.tsx` |
| **TypeScript 工具文件** | kebab-case + `.ts` | `retry-manager.ts` |
| **目录** | kebab-case，全小写 | `components/glass/` |
| **React 组件名** | PascalCase，与文件名一致 | `GlassComponent` |
| **函数/变量** | camelCase | `calculateScore()` |
| **常量** | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| **接口** | PascalCase + `I` 前缀（可选，与项目约定一致） | `Inspiration` / `IInspiration` |

### 修复步骤

1. **建立规范**：在 `docs/CODE_STANDARDS.md` 中明确上述约定
2. **统一入口文件**：每个目录使用 `index.ts` 作为统一导出点
3. **路径别名**：在 `tsconfig.json` 中配置路径别名，import 使用别名而非相对路径
4. **大小写审查**：Reviewer 检查 import 路径大小写是否与实际文件名完全匹配

### tsconfig.json 路径别名示例

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@components/*": ["src/components/*"],
      "@screens/*": ["src/screens/*"],
      "@services/*": ["src/services/*"],
      "@store/*": ["src/store/*"],
      "@types/*": ["src/types/*"]
    }
  }
}
```
